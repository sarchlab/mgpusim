## Change from v1

In v1, stencil2d and fft benchmarks were running with default multi-iteration/pass settings that don't match the single-kernel-launch measurement used in the real hardware reference. Specifically:
- **stencil2d**: was running with `-iter 5` (default), now `-iter 1` → times dropped ~3.3-3.5x
- **fft**: was running with `-passes 2` (default), now `-passes 1` → times dropped ~1.5-1.6x

**Fix applied:** The default values for `-iter` (stencil2d) and `-passes` (fft) have been changed in the source code to 1, matching the real hardware measurement methodology. Re-running these benchmarks with default flags will now produce v2-consistent results.

## Accuracy Tiers

### ✅ Good Accuracy (avg |error| < 50%) — 5 kernels, 31 points
- **matrixmultiplication** — 4.8% avg error. All points within 10%. Compute-bound, very well-modeled.
- **bicg** — 20.2% avg. Regular memory access. Simulator consistently "too fast."
- **matrixtranspose** — 34.5% avg. Moderate constant overhead relative to small kernel sizes.
- **atax** — 40.4% avg. Simulator "too fast" — under-models real overhead.
- **fastwalshtransform** — 45.5% avg, though error grows at larger sizes (97% at 65536).

### ⚠️ Marginal Accuracy (50-100%) — 3 kernels, 19 points
- **fir** — 58.1% avg. Small sizes are good (23-27%); large sizes degrade (154% at 262144).
- **stencil2d** — 61.8% avg. Significantly improved from v1 (was 439%). Still ~50-112% too slow. The simulator overestimates memory latency for the stencil pattern.
- **vectoradd** — 87.7% avg. Small sizes OK (14-33%); large sizes severely overestimate (381% at 524288).

### ❌ Poor Accuracy (>100%) — 2 kernels, 12 points
- **fft1D_512** — 102.0% avg. Improved from 218% but still roughly 2x too slow. Multiple kernel dispatch overhead likely a factor.
- **relu** — 106.8% avg. Same pattern as vectoradd — dominated by large-size streaming errors.

## Error Analysis

### Pattern 1: Large-size streaming workloads (vectoradd, relu, fir)
Error grows with problem size. At small sizes (≤16384 elements), error is 14-35%. At large sizes (≥65536), error exceeds 50% and grows rapidly. The simulator's memory subsystem appears too slow for simple streaming access patterns — possibly the memory instruction issue rate or DRAM throughput is undermodeled.

### Pattern 2: Kernel dispatch overhead
For very small, fast kernels (vectoradd/relu at small sizes, matrixtranspose), there's a roughly constant overhead of ~2-4μs that the simulator adds. This is the kernel launch/completion overhead modeling.

### Pattern 3: Compute-bound kernels work well
matrixmultiplication, bicg, atax — all have high arithmetic intensity and the compute model (SPU=16) provides accurate timing.

### Pattern 4: FFT still elevated
Even with -passes 1, fft1D_512 is ~2x too slow. This kernel has complex memory access patterns (butterfly pattern) that may stress the memory subsystem differently. The 114% error at 131072 elements suggests the sim's per-workgroup overhead is too high.
